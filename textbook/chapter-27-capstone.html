    01 // what an agent actually is

    What an agent actually is

    2. The five components below are the same five Anthropic named in Building Effective Agents (December 2024). Each is a unit a production team budgets, version-controls, and pays for separately, which is why a capstone scoped at the component level survives a teammate leaving or a model upgrade arriving mid-semester.

    Ch 1's agent primitive decomposes into the five pieces a marketer edits, debugs, and budgets separately.

    Component 1: the prompt

    The prompt has two slots. The system prompt is the stable part: who the agent is, what it must always do, what it must never do, brand voice, banned words, audience. Cached at 90 percent off, effectively free after the first call. The user prompt is run-specific: this Monday's competitor list, this draft to lint, this question to answer.

02 // tools and tool-calling

    Tools and tool-calling, in detail
    


    4. A tool is a function with a name, parameters, and a return value. The model never executes it; it only emits the request the runtime runs.
    


    You wrote read_url(url) in Ch 1 as the canonical first tool. Each tool description is a JSON Schema object naming the tool, what it does, and the arguments it accepts with their types, as in the schema below, and the return value comes back as a tool result message.
    


    {
  "name": "search_competitors",
  "description": "Returns the top 10 brands in a category with weekly review velocity",
  "parameters": {
    "type": "object",
    "properties": {
      "category": { "type": "string", "description": "e.g., insulated drinkware" },
      "lookback_days": { "type": "integer", "default": 7, "minimum": 1, "maximum": 30 }
    },
    "required": ["category"]
  }
}
    


    The principle is that the better the description, the better the model's tool selection: a vague description ("searches for stuff") misses, while a specific one arrives at the right tool with the right parameters on the first try.

mcp = FastMCP("brand-voice")

@mcp.resource("brandvoice://styleguide")
def styleguide() -> str:
    return open("styleguide.md").read()

@mcp.tool
def lint_brand_voice(text: str) -> dict:
    banned = ["leverage", "synergy", "unlock", "—"]

    return f"Rewritten for {audience}: {text}"

if __name__ == "__main__":
    mcp.run()

    Reflection, evals, and the eval harness
    


    6. An eval is one test case (input, expected output, scoring rule). An eval set is a collection. An eval harness is the code that runs the set and produces a summary comparable across model + prompt versions.
    


    An agent without reflection produces confident garbage when something subtle is wrong, which is most of the time. The fix is the eval harness: a fixed test set the agent is run against and scored on.

  Eval harness
  The code (or platform feature) that runs a fixed set of test cases against the agent on every change and scores each output. Three layers stack (golden set, LLM-as-judge, human spot-check), and none alone is enough.

    


    Layer 1: the golden set
    


    A golden set is a small library of test inputs the team has paired by hand with the output it has decided is correct, like a sample exam with an answer key. Launch with about five cases, grow to thirty by ship time, and reach about a hundred a year into production, using the Anthropic Console's Evaluations tab to handle the run-on-every-change wiring. Writing the cases is itself the work of deciding what the agent is for, since a team that cannot write five concrete cases has not yet decided what the agent does.

    Layer 2: LLM-as-judge

    Failing runs should teach the agent to do better, which is the Reflexion architecture (Shinn et al., 2023): the agent reads its failing run, writes a paragraph of critique, and loads it back on the next attempt, so the text in a small JSON-lines file is the entire learning signal. The capstone scales it to one eval folder, where a separate Sonnet 4.6 call reads each failing trace, writes a 150-200-word critique tagged with one of five failure modes, appends it to .evals/reflections.jsonl, and every subsequent run reads back the last three entries. The block below extends the run_evals.py file Waypoint 8 builds.
    



    # reflexion.py — extend run_evals.py with a critique writer
import os, json, datetime
from anthropic import Anthropic
client = Anthropic()

CRITIC_SYSTEM = (
    "You are the Self-Reflection model in a Reflexion loop. "
    "Read the failing run trace and the Evaluator's score. "
    "Write a 150-200 word critique explaining what went wrong "
    "and what the Actor should do differently next time. "
    "End with one tag on its own line: LEADING / WRONG-TOOL / "
    "MISSING-CONTEXT / OVERCONFIDENT-CLAIM / FORMAT-DRIFT."
)

def write_critique(run_id, trace, rubric_score):
    """Step two + three: critique + append to reflections.jsonl."""
    critic = client.messages.create(
        model="claude-sonnet-4-6",
        system=CRITIC_SYSTEM,
        max_tokens=400,
        messages=[{"role": "user", "content":
            f"RUN TRACE:\n{trace}\n\nRUBRIC SCORE:\n{rubric_score}"}],
    )
    critique = critic.content[0].text
    tag = critique.strip().split("\n")[-1]
    entry = {
        "timestamp": datetime.datetime.now().isoformat(),
        "run_id": run_id,
        "tag": tag,
        "critique": critique,
    }
    with open(".evals/reflections.jsonl", "a") as f:
        f.write(json.dumps(entry) + "\n")
    return entry

def last_three_critiques():
    """Step four: read the last three entries to inject into next run's system prompt."""
    if not os.path.exists(".evals/reflections.jsonl"):
        return ""
    lines = open(".evals/reflections.jsonl").read().strip().split("\n")
    last = [json.loads(ln) for ln in lines[-3:]]
    section = "CRITIQUE OF LAST RUNS\n"
    for e in last:
        section += f"- [{e['timestamp']} / {e['tag']}] {e['critique']}\n"
    return section

      # run_evals.py
import os, json, glob
from dotenv import load_dotenv
from anthropic import Anthropic

load_dotenv()
client = Anthropic()  # reads ANTHROPIC_API_KEY from .env

SYSTEM = open("CLAUDE.md").read()  # the brand-voice system prompt

passes = 0
fails = 0
for path in sorted(glob.glob("evals/case-*.json")):
    case = json.load(open(path))
    # Step A: generator
    gen = client.messages.create(
        model="claude-haiku-4-5",
        system=SYSTEM,
        max_tokens=512,
        messages=[{"role": "user", "content": case["input"]}],
    )
    output = gen.content[0].text
    # Step B: judge
    judge_prompt = (
        f"PASS CRITERIA: {case['pass_criteria']}\n\n"
        f"AGENT OUTPUT:\n{output}\n\n"
        "Respond with exactly one word: PASS or FAIL."
    )
    judge = client.messages.create(
        model="claude-haiku-4-5",
        max_tokens=8,
        messages=[{"role": "user", "content": judge_prompt}],
    )
    verdict = judge.content[0].text.strip().upper()
    print(f"{path}: {verdict}")
    if "PASS" in verdict:
        passes += 1
    else:
        fails += 1
print(f"\n{passes}/{passes+fails} passed.")

# Brand-voice lint

Voice constitution lives at references/styleguide.md. Read it first.

## Rules
1. Three voice attributes (load-bearing): irreverent, blunt, anti-corporate. Examples in the styleguide.
2. Banned words: leverage, synergy, unlock, delve, navigate, landscape, crucial, paradigm.
3. Audience: 22-to-34 DTC drinkware buyer. Write for the suit, not the drone.
4. Sentence patterns: short. Declarative. Concrete noun before abstract claim.

## Procedure
1. Read the draft.
2. Tag every banned word with [BANNED: word] inline.
3. Score each sentence 1-5 on voice fit. Flag any sentence below 4.
4. Return ISSUES (one bullet per flagged item) then NEUTRAL REWRITE (the full draft, fixed).
5. End with one line: cost-of-this-run in tokens and dollars.

## Reflection guard
Before shipping, run the eval set at evals/voice-pass.json. If any case fails, revise once and re-run. After two failures, escalate to a human reviewer.

    Check Your Understanding
    

    Answer each in your head, then click to check.
    

    1. The chapter says a team should scope its capstone at the component level rather than the task level. What are the five components, and why does the level matter?
    The five components are the prompt (a stable system slot plus a fresh user slot), the tools the model calls between thinking turns, the memory that lives outside the model, the planning that decomposes a goal into tool calls, and the reflection that grades the output against a rubric. The level matters because each component is a unit a team edits, debugs, and pays for separately, so a project scoped by component keeps running when a teammate leaves or a model upgrade lands mid-semester, while a project scoped only as a task has no clean seam to repair when one piece breaks.
    

    2. Why does an agent need a memory layer at all when the underlying model is so capable, and what are the three tiers?
    The model is stateless, so its weights are frozen and it forgets everything at the end of every turn, which means anything the agent must carry between calls has to live outside the model and be loaded back into the next prompt. The three tiers run from cheapest to richest: a CLAUDE.md text file the runtime reads at session start, a pinned folder the agent retrieves from on demand through the RAG pattern, and a knowledge-graph server that stores accumulated facts as entities and relations. The accumulation is why a weekly agent's third run starts the morning already knowing what its first two runs discovered.
    

    3. State the M-by-N argument for the Model Context Protocol, and explain why it made MCP a standard so fast.
    Without a shared standard, every pairing of an AI app with a tool needs its own custom adapter, so ten apps wired to fifty tools costs roughly five hundred pieces of glue. With MCP, each app and each tool speaks the protocol exactly once, which collapses the cost to about sixty. That single combinatorics argument is the cleanest one-sentence justification for any standard, and it made MCP the de-facto standard within twelve months of release, after which governance moved to the Linux Foundation's Agentic AI Foundation, the institutional signal a CIO reads to mean the protocol will outlive any one vendor.
    

    4. A team is excited about scheduled agents and wants its capstone to run unattended on a cron timer from week one. Why does the chapter call this a mistake?
    The four rungs are a ladder rather than a menu, and each rung adds exactly one thing the rung below cannot: a Skill adds reusable instructions, an MCP server adds live data, and a schedule removes the human from the loop. A team that jumps to a Tier 4 scheduled agent before writing the Tier 2 Skill ends up hard-coding brand context inside the agent's system prompt, where it cannot be shared, version-controlled, or audited. The decision rule is mechanical: climb only when the next rung adds something the current one cannot, which for most capstones settles the agent at Tier 3 or Tier 4 but never by skipping the rungs underneath.
    

    5. The chapter insists the eval set report pass^k rather than pass@1. What is the difference, and why does it change which agents look shippable?
    Pass@1 is the chance of success on a single attempt, while pass^k is the chance of success on k consecutive attempts at the same task, which measures reliability rather than peak capability. The difference is large: Sierra Research's tau-bench found top function-calling models cleared pass^8 below 25 percent on retail customer-service tasks even when their single-attempt scores crossed 86 percent. An agent that succeeds nine times out of ten sounds shippable on a one-shot read, but its chance of eight clean runs in a row collapses, so reliability does not equal capability and the eval set has to report the consecutive-run number.
    

    6. The chapter claims output quality is a product of the rubric, not the model. What is the evaluator-optimizer loop, and how does that claim follow from it?
    The evaluator-optimizer pattern is one model generating a draft and a second model call scoring it against a written rubric, so when the draft fails the critique becomes the next prompt and the loop runs until the output passes or hits an iteration limit. The claim follows because swapping the rubric makes the same generator return a different shape of output, and tightening the rubric drops the pass rate until the generator is forced to solve the problem the rubric describes. That is why the brand-voice Skill carries banned-word lists and sentence-pattern rules rather than free-form prose, since the Skill is the judge's reference text on every pass.
    

    7. Name three of the six situations where the chapter says a marketer should not build an agent, and give the safer alternative for each.
    A single-shot task wants a chat session, because the break-even on packaging a prompt as a Skill is around the third repetition and below that the agent is overhead with extra steps. A low-trust write-enabled domain wants human-in-the-loop on every destructive call, because a malicious page read through a fetch tool can inject instructions that trigger a destructive action, and once the human checkpoint is in place the system is really a workflow rather than an autonomous agent. A sunsetting platform wants a current tool, since building on the OpenAI Assistants API or Project Mariner means building on something already scheduled for shutdown. The hard default behind all of these is read-only.
    

    8. METR measured a frontier model's autonomous task horizon at 320 minutes in early 2026. Why does the chapter tell a capstone team to ship at roughly half that length?
    The 320-minute figure is the point at which the model finishes an expert-length task at 50-percent reliability, which means it is as likely to fail as to succeed, so it is not a shippable threshold. The shippable band of 80-to-90-percent reliability falls at roughly half the 50-percent length, which in May 2026 is about two and a half hours of autonomous work between human checkpoints. The chapter also warns the horizon is not flat across task type, since GUI-driving tasks lag API-driving tasks by a wide margin, and the doubling rate near four months means the team has to re-baseline the ceiling every semester rather than trust last year's number.
